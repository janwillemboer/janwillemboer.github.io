3. Quality Management & Deployability

This track is about quality management, testing, and CI/CD.

Companies often have processes to ensure quality and to enforce that releases are tested before going to production. In this track you build a pipeline around your product that has rigid quality management. This includes a DTAP-environment (or at least an AP-environment).

  • There is a linter in place (or other static code analysis tool)
  • Unit tests cover all code
  • git commits can never go directly to the master branch. Changes can only be merged from separate branches. Merges can only be done if they have been reviewed, if there are no linter errors, if the product builds, and if all unittests succeed
  • After a build, the changes from the master branch are automatically deployed to an acceptance environment.
  • After deployment, all API and/or UI-tests are done automatically. Note that for this to work, tests have to be idempotent, meaning they are independent and can always be repeated with the same result.
  • After success and after manual approval (triggered by some signal to the admin) the deployment is done on the next environment.
  • Each environment has a totally separate stack.
  • Database changes and dataconversion are also done automatically.
  • Documentation. All measures taken, and all features in your advanced track should be described in detail in the technical design, including, of course, all configuration details. A manual should provide information to admins on the CI/CD system. A show-and-tell document proves that the system works.

Basic requirements for grading

  • There is a working automatic deployment using GitLab CI.
  • There is a show-and-tell document to prove the system works.
